**Gemini CLIは、ただの「ターミナルで使えるGemini」ではなくなりました。**

2026年4月末のv0.40系アップデートで、記憶、タスク管理、ローカルGemma、MCPリソース、セキュリティ強化が入り、**Claude CodeやCodexと比較すべき開発エージェントに近づいています**。

アップデートはターミナルで以下のコマンドを実行します。

```text
npm install -g @google/gemini-cli
```

# Gemini CLIは何が変わったのか

![画像](https://pbs.twimg.com/media/HHMZd7EaYAA0QpO?format=jpg&name=large)

結論から言うと、Gemini CLIは「質問に答えるCLI」から「**作業を継続するCLI**」へ進んでいます。

GoogleのGemini CLIは、もともとオープンソースのターミナル向けAIエージェントです。

公式READMEでは、Googleアカウントで使える無料枠、Gemini 3モデル、1Mトークン級のコンテキスト、Google Search grounding、ファイル操作、シェル実行、MCP対応が主な特徴として説明されています。

ここまでは、従来のAI CLIと同じ文脈で理解できます。コードを読ませる。修正させる。コマンドを実行させる。外部情報を検索させる。

**ただしv0.40.0では、そこに「継続性」が加わりました。**

リリースノートでは、メモリ、スキル抽出、MCPリソース、通知、UI整理、サンドボックス、コマンド検証など、多数の改善が並んでいます。

重要なのは、Gemini CLIが一回の会話だけで完結する道具ではなく、プロジェクト文脈を保存し、作業の進捗を持ち、外部ツールと接続しながら動く方向へ進んだことです。

# 一番大きい変化はメモリです

![画像](https://pbs.twimg.com/media/HHMZoSma4AAxDHT?format=jpg&name=large)

今回のアップデートで一番大きいのは、メモリの考え方です。

Gemini CLIは、プロジェクトの前提や運用ルールをMarkdownファイルとして扱う方向へ進みました。

v0.40.0のリリースノートには、従来のMemoryManagerAgentを置き換え、**4階層のメモリをプロンプト駆動で編集する変更**が入っています。

この4階層は、実務上かなり合理的です。

プロジェクト全体のルールは、リポジトリ直下の\`GEMINI.md\`に置く。特定ディレクトリだけのルールは、その配下の\`GEMINI.md\`に置く。個人のローカル設定は、コミットしないメモリに置く。横断的な個人の好みは、グローバルの\`~/.gemini/GEMINI.md\`に置く。

これはAIエージェントの使い勝手を大きく変えます。

毎回「このプロジェクトではこう書いてください」「このディレクトリだけ例外です」「ローカルではこのコマンドを使います」と説明し直す必要が減ります。しかも記憶がMarkdownとして残るため、人間が読めます。レビューできます。チームで共有できます。

ここが重要です。

AIの記憶がブラックボックスのままだと、便利な反面、何を覚えているのか分かりません。Gemini CLIの方向性は、記憶を“見えるファイル”にすることです。これは開発現場に向いています。

# Auto Memoryはスキル化の入口です

もう一つの注目点が、Auto Memoryです。

v0.40.0では、過去セッションから再利用できる作業パターンを抽出するための変更が入りました。リリースノートには、スキル抽出エージェント、\`autoMemory\`フラグ、JSONLセッションログ対応などが含まれています。

![画像](https://pbs.twimg.com/media/HHMZvYTasAACh1w?format=jpg&name=large)

要するに、Gemini CLIは**「この人は何度も同じ作業をしている」「この修正手順は再利用できる」**と判断し、スキル候補として残す方向へ進んでいます。

これは**Claude CodeのSkillsに近い発想**です。

Claude Codeでは、Skillsは特定の作業手順や知識を再利用する仕組みです。一方、Gemini CLIのAuto Memoryは、過去の会話や作業履歴から候補を作る点に特徴があります。

もちろん、ここはまだ実験的です。自動抽出されたスキルをそのまま信じるのではなく、人間がレビューして採用する前提で見るべきです。

ただ、方向性はかなり強いです。AIエージェントは、単に賢いモデルを呼ぶだけでは限界があります。実務では**「自分の作業の癖」「チームの手順」「過去にハマった修正」**が効きます。Auto Memoryは、そこを資産化しようとしています。

# Gemma対応でローカル実行への道が見えた

![画像](https://pbs.twimg.com/media/HHMZ1E2agAA1GNC?format=jpg&name=large)

Gemini CLI v0.40.0では、\`**gemini gemma**\`によるローカルモデルセットアップが追加されました。v0.41系のプレビューでは、Gemma 4モデル対応も進んでいます。

現時点で、これは「Gemini CLIがすべてローカルで完結するようになった」という意味ではありません。公式リリース上では、Gemma対応は実験的な位置づけです。ローカルGemmaは、まず補助的な用途から始まっている段階です。

しかし、この一歩は大きいです。

クラウドの大きなモデルは強力です。複雑な設計、長いコード理解、精度の高い推論には向いています。一方で、軽い判定、ルーティング、短い要約、単純な補助処理まで全部クラウドに投げる必要はありません。

将来的に、**軽い処理はローカルGemma、重い処理はGemini、外部情報はGoogle Search、ツール連携はMCP**という構成になると、Gemini CLIはかなり面白い立ち位置になります。

Claude CodeやCodexと比べたとき、Gemini CLIの独自性はここにあります。Google Search、Google Cloud、Gemini、Gemmaを同じエージェント体験の中に置けることです。

# Task Trackerで長い作業に耐える

![画像](https://pbs.twimg.com/media/HHMZ6XtbwAA6VFo?format=jpg&name=large)

AIエージェントの弱点は、長い作業で進捗がぼやけることです。

最初は「テストを直して」「UIを改善して」「READMEも更新して」と依頼していても、途中で枝分かれします。

調査が入り、修正が入り、追加の不具合が見つかり、最後に何が残っているのか分かりにくくなります。

Gemini CLI v0.40.0では、**experimental task trackerのドキュメントが追加**されています。これは、複雑な目的を内部タスクとして持ち、進捗を追いやすくするための機能です。

地味ですが、実務では重要です。

AIエージェントに求められるのは、単発回答ではありません。調査して、計画して、直して、確認して、残件を潰すことです。そのためには、タスクの見える化が必要です。

ここはCodexやClaude Codeも重視している領域です。Codexはローカルやクラウドでタスクを進め、Claude CodeはSubagentsやAgent teamsで作業を分担します。Gemini CLIはTask Trackerによって、まず単一セッション内の進行管理を強化していると見てよいです。

# Claude Codeとの違い

Claude Codeは、開発エージェントとしての運用設計が非常に強いです。

公式ドキュメントでは、**\`CLAUDE.md\`、Skills、MCP、Subagents、Hooks、Plugins、Agent teams**が整理されています。

\`CLAUDE.md\`は常時読み込む文脈、Skillsは再利用可能な手順、MCPは外部サービス接続、Subagentsは独立した専門ワーカー、Hooksは決定的な自動処理です。

つまりClaude Codeは、エージェントをどうチームの作業環境に組み込むかまで設計されています。

Gemini CLIも今回のアップデートで近い領域に入りました。\`GEMINI.md\`階層は\`CLAUDE.md\`に近く、Auto MemoryはSkillsに近く、Task Trackerは長時間作業の管理に近い役割を持ちます。

ただし、まだ差があります。

Claude CodeはSubagentsやHooks、Pluginsの運用が強いです。専門エージェントに調査させる。レビューだけを任せる。編集後に自動でlintを走らせる。チームで配布できる形にする。こうしたエージェント運用の完成度では、Claude Codeが先行しています。

Gemini CLIの強みは別の場所にあります。

Google Searchで外部情報に強い。Geminiの大きなコンテキストを使える。Google CloudやCode Assistとの接続がある。さらにGemmaによるローカル活用の方向が見えている。

Claude Codeが「開発ワークフローの運用基盤」だとすれば、Gemini CLIは「Googleのモデルと検索とローカルモデルをつなぐCLIエージェント」に近づいています。

# Codexとの違い

OpenAI Codexは、コード作業の委任に強いエージェントです。

Codex CLIはローカルで動くコーディングエージェントとして提供されています。公式READMEでは、npmやHomebrewで導入でき、ChatGPTプランでも利用できると説明されています。さらにOpenAIのドキュメントでは、CodexはWeb、IDE、CLI、SDK、CI/CDで使えるコーディングエージェント群として説明されています。

ここがGemini CLIとかなり違います。

Codexは、ローカルCLIだけではありません。Codex Cloudでは、専用のクラウド環境でバックグラウンドの作業を進められます。PR作成、コードレビュー、修正、テスト追加など、実装委任の体験が強いです。

一方、Gemini CLIは、今のところターミナル体験を中心に進化しています。Google Search、MCP、メモリ、Gemma、Task Trackerを足して、ローカル作業環境の中で賢くなる方向です。

ざっくり言うと、Codexは「**コード作業を任せる**」体験が強いです。Claude Codeは「**開発ワークフローを組む**」体験が強いです。Gemini CLIは「**検索、巨大文脈、Google連携、ローカルGemmaを束ねる**」方向に強みがあります。

どれが上かではありません。使いどころが違います。

実装を一気に任せたいならCodex。チームの作業手順や専門エージェントを作り込みたいならClaude Code。調査、長文脈、Google連携、オープンソース性、ローカルモデルの将来性を重視するならGemini CLIです。

# 今回のアップデートで見るべきポイント

今回のGemini CLIアップデートで見るべきポイントは、機能の数ではありません。

見るべきは、**方向性**です。

AIエージェントは、モデル単体の賢さだけでは差がつきにくくなっています。これから差が出るのは、文脈をどう保存するか、作業をどう分担するか、外部ツールとどうつなぐか、ローカルとクラウドをどう使い分けるかです。

**Gemini CLI v0.40系は、この4つに同時に手を入れています。**

メモリは\`GEMINI.md\`階層へ。作業パターンはAuto Memoryへ。進捗管理はTask Trackerへ。ローカル処理はGemmaへ。外部連携はMCPリソースへ。安全性はコマンド検証、allowlist、ワークスペース信頼、サンドボックス強化へ。

これは、Gemini CLIがClaude CodeとCodexを本気で意識しているということです。

まだ実験的な機能も多いです。Auto MemoryもTask TrackerもGemmaも、現時点では過信すべきではありません。実務で使うなら、設定を理解し、生成されたメモリやスキル候補をレビューし、コマンド実行権限を慎重に扱う必要があります。

それでも、今回のアップデートでGemini CLIの見え方は変わりました。

「無料枠が大きいGeminiのCLI」ではなく、「**記憶を持ち始めたGoogle系開発エージェント**」として見るべきです。

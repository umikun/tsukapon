ペーストされた文章（またはエディタ・ブラウザ選択）の内容を深掘り・調査し、背景・関連概念・最新動向・批判的視点まで網羅した「調査ノート」をマークダウン形式で出力するスキルです。

## 用途

ニュース記事・ツイート・書籍の引用・論文抜粋・クライアント資料など、「読んだけど腹落ちしていない」テキストを投入すると、
WebSearch / WebFetch を使って事実確認と周辺情報の補強を行い、1枚のマークダウン調査ノートに仕上げる。

## 入力形式（3パターン対応）

**パターン A: 直接貼り付け**
```
/deep-dive 調査したい本文をそのまま貼り付け
（複数段落OK。URLが含まれていてもよい）
```

**パターン B: エディタ選択**（`<editor_selection>` タグがある場合）
- 引数が空なら `<editor_selection>` の内容を対象にする
- 対象ファイルパスも記録（調査ノートから `出典:` としてwikilinkで参照）

**パターン C: ブラウザ選択**（`<browser_selection>` タグがある場合）
- 引数が空なら `<browser_selection>` の内容を対象にする
- `url` / `title` 属性があれば冒頭の「出典」欄に記録する

※ 3パターンすべて該当しない・引数も空の場合は、「深掘り対象のテキストを貼り付けてください」と聞き返す。

## 処理手順（8ステップ）

### Step 1: 対象テキストの取得と正規化

- 入力パターンを判定し、対象テキスト `TEXT` を1つに確定させる
- ファイル名用の `topic_slug` を生成（後述の命名規則）
- 出典情報（ファイルパス／URL／タイトル）があれば記録

### Step 2: 構造解析

TEXT を読み解き、以下を抽出してメモする：

- **主題 (Main Topic)**: 1文で要約
- **キーワード / 専門用語**: 初見・略語・固有名詞を5〜15個程度
- **主張・論点 (Claims)**: 著者が明示的に述べている断言・数字・因果関係
- **前提 / 暗黙の仮定 (Hidden Assumptions)**: 明言されていないが成り立っていそうな前提
- **疑問点 (Open Questions)**: 読んでいて詰まったポイント、確証が欲しい数字・引用
- **時制情報**: 日付・年月が含まれているか（古い情報ならフラグ）

### Step 3: 調査計画の策定

抽出した「疑問点 + キーワード + 主張」から、WebSearch / WebFetch に投げるクエリを3〜6本組み立てる。
以下の観点をカバーすること：

1. **背景・歴史** — 概念がいつ・誰から・どんな文脈で生まれたか
2. **最新動向（現在日付基準で6ヶ月以内）** — 2026年時点で情報が更新されていないか
3. **主張のファクトチェック** — TEXT 内の具体的数字・引用の裏取り
4. **反対意見・代替案** — 批判・ポジショントーク・別解釈の存在
5. **関連概念 / 隣接分野** — 一緒に学ぶと理解が深まるテーマ

### Step 4: 外部情報の収集

- **WebSearch**: 一次情報（公式サイト・論文・一次ソースメディア）を優先してURLを取得
- **WebFetch**: 取得したURLから本文を読み、主張の裏取りと最新日付の確認を行う
- 取得した情報には **必ずURLと取得日** をメモ（後段の「参考資料」に記載）
- 情報が古い（1年以上前）・ソースが不明瞭な場合はその旨も記録する

**禁止事項**:
- ソースURLのない憶測情報を「事実」として書かない
- 自分の内部知識（2025年以前）だけで最新動向を断定しない（必ずWeb確認する）

### Step 5: 調査ノートの組み立て

以下のテンプレートに沿って `調査/YYYY-MM-DD-{topic_slug}.md` を生成する。

```markdown
---
created: YYYY-MM-DD
tags: [調査, {主要キーワード1}, {主要キーワード2}]
source: {URL or vault内パス or "直接貼付"}
action: {投稿ネタ|運用参考|サブスクネタ|採用なし}
---

# {主題を1行で}

> **TL;DR**
> {3〜4行で「結論」「キモになる事実」「読者が次にとるべき行動/理解」を要約}

---
```

> 🆕 **2026-05-07 frontmatter拡張**: `action` プロパティを必須化。Bases / Dataview から「投稿ネタ」「運用参考」「サブスクネタ」だけを横断抽出できるようにする。**メモセクションの内容を踏まえてClaudianが判定し、frontmatterにセット**する。

#### `action` プロパティの判定ルール

| 値 | 採用基準 | 典型例 |
|---|---|---|
| `投稿ネタ` | SNS投稿（X単発／スレッド／Threads／note記事）に転用できる「一般読者ウケしそうな」題材 | 新ツール紹介、AI話題、業界トレンド |
| `運用参考` | 自分のSNS運用・vault運用・スキル設計の改善ヒントになる題材（外部発信より自分の運用最適化向き） | 他者の運用ノウハウ、ツール比較、ワークフロー設計論 |
| `サブスクネタ` | Day Oneサブスク・有料note等の **クローズドな有料コンテンツ向け**（無料枠で出すには濃すぎる or プレミア感を持たせたい） | 個人体験ベースの深い検証、年収・収益・私的なノウハウ |
| `採用なし` | 調査はしたが、現時点で発信・運用どちらにも使い道が見えない（情報整理だけで終了） | 専門外、既出すぎる、ターゲット読者と乖離 |

**判定の優先順位**:

1. メモセクションの「このネタの使い道」記述を最優先で参照
2. SNS投稿として刺さりそう（フック性・新規性）→ `投稿ネタ`
3. 自分の運用にすぐ反映できる学び中心 → `運用参考`
4. 個人的・濃密で有料コンテンツ向け → `サブスクネタ`
5. どれにも当てはまらない／メモセクションが「特になし」 → `採用なし`

**`等` の扱い**: 上記4値で迷ったら、メモを根拠に1つ選ぶ。複数該当する場合は **「主たる用途」一つだけ** を入れる（複数値は入れない、Bases絞り込みが効かなくなるため）。**他の値は使わない**（4値固定）。

### Step 6: 保存と出力

- ディレクトリ `調査/` が存在しなければ作成する（`mkdir -p 調査`）
- ファイル名: `調査/YYYY-MM-DD-{topic_slug}.md`
  - 同日同slugが既存なら末尾に `-2`, `-3` を付けて衝突回避
- Write ツールで保存

### Step 8: 自己検証

提出前に以下をチェック：

- [ ] 「最新動向」セクションの各項目にURLと年月が付いているか
- [ ] 「参考資料」にソースURLが最低3本あるか
- [ ] 「主張のファクトチェック」で 🔍 未確認 が半分以上になっていないか（少なすぎる場合は Step 4 をやり直す）
- [ ] 元テキストを曲解していないか（引用した主張が元意と一致しているか）
- [ ] AIっぽい硬い結び（「いかがでしたか」「活用してみましょう」等）が混じっていないか

1項目でも欠けていたら、その部分だけ追加調査して補完してから保存する。

## 命名規則（topic_slug）

- 主題を英数字＋ハイフンで10〜30文字に圧縮
- 日本語テーマでも **英語スラッグを優先**（ファイル検索しやすいため）
  - 例: 「MCP Server の最新動向」 → `mcp-server-trends`
  - 例: 「日銀のマイナス金利解除」 → `boj-negative-rate-exit`
- 英訳が難しい固有名詞はローマ字でOK（例: `obsidian-canvas`）

## 制約事項

- **出力は必ず1つのマークダウンファイル**。分割しない
- **ソースURLのない断定は書かない**（内部知識で補うときは「一般に〜と言われる」等のヘッジ語を使う）
- Obsidian独自記法（`[[]]` `![[]]`）を活用してvault内の関連ノートへリンクする
  - 特に `Clippings/` や既存の `SNS運用/` 内に関連ノートがありそうなら検索してリンクを張る
- **チャット内だけで完結させない**。必ずファイル保存する
- トーンは「気の利いた先輩が調べてくれたメモ」くらい。講義調・AI定型文は避ける

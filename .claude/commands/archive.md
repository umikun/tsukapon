完了済みのノートを「アーカイブ済み」に自動仕分けし、タイムスタンプ付きの集約ログ `archive.md` に追記するスキル。Hermes Kanbanの「Smart Card Archive」をObsidian vault運用に持ち込んだもの。

## 用途

- 投稿済みの日次X投稿（`SNS運用/post/day*.md`）を `_fin/` に逃がしてフォルダを軽く保つ
- 完了したnote記事・スレッド・調査ノートを「物理的に隔離」しつつ、いつ何が完了したかを `archive.md` で時系列追跡
- 任意のフォルダに対して「完了マーカー → 自動移動 + 時刻ログ」を1コマンドで適用

## 入力形式（4パターン対応）

**パターン A: フォルダ指定（最も標準）**
```
/archive SNS運用/post
```
→ そのフォルダ内で完了マーカーが付いたファイルを抽出して処理。

**パターン B: 単一ファイル指定**
```
/archive SNS運用/post/day60.md
```
→ そのファイルだけをアーカイブ対象に。

**パターン C: 引数なし（対話モード）**
```
/archive
```
→ 主要な運用フォルダ（`SNS運用/post`, `SNS運用/post/thread`, `SNS運用/post/article`, `SNS運用/note`, `調査`）を一覧表示し、ユーザーに選ばせる。

**パターン D: ドライラン**
```
/archive SNS運用/post --dry-run
```
→ 実際には移動せず、「何が動くか」だけプレビュー表示。

## 完了マーカーの判定ルール（OR条件）

以下のいずれかを満たすファイルを「完了済み」と判定する。

1. **frontmatter フラグ**:
   - `archived: true`
   - `posted: true`
   - `投稿済み: true`
   - `status: done` / `status: 完了` / `status: posted`
2. **本文中のHTMLコメント**:
   - `<!-- ARCHIVE -->` または `<!-- DONE -->` がファイル末尾に存在
3. **ファイル名規則**（フォルダ依存・任意）:
   - `SNS運用/post/day*.md` で本文先頭が「投稿済み」「✅」で始まる場合

判定に迷う場合（マーカーが曖昧・部分一致）は、ユーザーに確認してから処理する。**勝手に動かさないこと**。

## 処理手順（7ステップ）

### Step 1: 対象フォルダ・ファイルの確定

- 入力パターンを判定して対象を確定
- パターンCの場合は候補一覧を表示してユーザーに選んでもらう

### Step 2: 完了マーカーのスキャン

- 対象フォルダ内の `.md` を1ファイルずつ読む（`Read` ツール）
- 完了マーカー判定ルールに合致するファイルだけリストアップ
- **以下は対象外**:
  - 既に `_fin/` `_archive/` 配下にあるファイル
  - 隠しファイル（`.` で始まる）
  - `archive.md` 自身

### Step 3: 移動先の決定

各対象ファイルについて、以下の優先順で移動先を選ぶ：

1. **同階層に `_fin/` フォルダが存在する** → `<folder>/_fin/` に移動（既存vault運用との互換）
2. それ以外 → `<folder>/_archive/YYYY-MM/` を作成して移動（年月で自動分類）

ファイル名はそのまま保持（既存ファイルと衝突した場合は末尾に `-2`, `-3` を付ける）。

### Step 4: 移動の実行

- `mv` で移動（**`rm` 禁止**。CLAUDE.mdルール準拠）
- 失敗したらその時点で停止して原因を報告（部分移動の状態を残さない）

### Step 5: archive.md への追記

対象フォルダ直下の `archive.md` に、移動したファイル全件を追記する。

`archive.md` が存在しない場合は新規作成（テンプレート下記）。
存在する場合は末尾に追記。

**フォーマット**:
```markdown
---
tags: [archive, auto-generated]
last_updated: YYYY-MM-DD HH:MM
---

# Archive Log — {フォルダ名}

> このファイルは `/archive` スキルが自動更新する。手動編集も可能だが、構造（## 日付見出し + - 時刻 + ファイル + 移動先）は崩さないこと。

## 2026-04-25

- **14:32** [[_fin/day60.md|day60.md]] — 投稿済みフラグ検出
- **14:32** [[_fin/day61.md|day61.md]] — `<!-- ARCHIVE -->` 検出

## 2026-04-24

- **18:05** [[_archive/2026-04/note-20260424.md|note-20260424.md]] — `posted: true`
```

**追記ルール**:
- 同日内なら同じ `## YYYY-MM-DD` 見出しの下に時刻順で追加
- 日付見出しがなければ作る（最新日付が上）
- ファイルへのリンクは**移動先の相対パス**で wikilink

### Step 6: 確認とクリーンアップ

- 移動先のファイル数 ＝ 対象ファイル数 が一致するか確認
- archive.md に追記された行数も確認
- 不一致なら `mv` 失敗の可能性 → 状態を報告

### Step 7: 完了報告

チャットに以下を表示：

```
✅ アーカイブ完了

📂 対象: {フォルダ名}
🗃 移動: N件
   → {移動先パス}
📝 archive.md に N行追記

ハイライト:
- [[移動先/file1.md]]
- [[移動先/file2.md]]
- ...

確認:
[[{folder}/archive.md]] を見ると全履歴が時系列で見られる。
```

## ドライランの場合の出力

```
🔍 ドライラン結果（実際には移動しない）

📂 対象: {フォルダ名}
✓ 検出: N件
   - file1.md ← `posted: true`
   - file2.md ← `<!-- ARCHIVE -->`
   ...

→ 移動先: {移動先パス}
→ archive.md に追記される行数: N

実行するなら `--dry-run` を外してもう一度。
```

## 制約・安全運用

- **`rm` は絶対に使わない**（CLAUDE.md準拠）。すべて `mv` で物理移動
- 1度に大量（20件以上）移動する場合は、ユーザーに「本当に全部やる？」を確認
- frontmatter は**書き換えない**（読むだけ）。「アーカイブ済み」の事実は archive.md 側に記録するため、元ファイルの内容は変えない
- `.obsidian/` `.claude/` `_ kiwami/tools/daily-log/` 配下は対象外（誤動作防止）
- 既存 `🔗 関連コンテンツ` callout は触らない（CLAUDE.md準拠）

## 想定ユースケース

| シーン | コマンド |
|---|---|
| 日次X投稿の完了処理（一日の終わりにまとめて） | `/archive SNS運用/post` |
| Xスレッドが投稿し終わったら | `/archive SNS運用/post/thread` |
| note記事が公開されたら | `/archive SNS運用/note SNS運用/note/note-20260425.md` |
| 何が動くか確認だけしたい | `/archive SNS運用/post --dry-run` |
| 全カテゴリまとめてリスト | `/archive`（対話モード） |

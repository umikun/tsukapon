note記事をリライト（品質改善・SEO最適化・内容充実化）し、続けてX記事転載版とXスレッド版まで一気に生成するスキルです。

## ⚠️ 必須パイプライン宣言（最重要）

このスキルは **Step 1 → Step 11 までを1コマンドで全て完了するパイプライン** です。以下のルールを厳守してください。

1. **Step 6（タグ選定 + 保存）の完了時点で絶対に最終応答を終えないこと。** Step 6 は中間マイルストーンであり、まだ Step 7・8・9・10・11 が残っています。
2. **Step 7（サムネイル文言生成）／Step 8（X Articles版の生成）／Step 9（Xスレッド版の生成）／Step 11（本日アクションプラン生成）は必須です。** 「オプション」「任意」ではありません。これら全ファイルが生成されるまでタスクは完了しません。
3. **最終出力は必ず5ファイル**: ①note本体、②サムネイル文言ファイル、③Article版、④Thread版、⑤本日アクションプラン。どれか1つでも欠けていれば未完了です。
4. **Step 10（完了検証と報告）で5ファイルの存在を確認してから**、最終応答を返してください。
5. 途中でユーザーに質問する／進捗確認を挟む必要はありません。**全Step完了までノンストップで走り切ってください。**

## 入力形式

記事ファイルのパスを1つ受け取ります：

```
記事ファイルパス
```

任意で、すでに公開済みの場合はnote記事URLを2行目に付与できます（Step 9のスレッド生成で使用）：

```
記事ファイルパス
note記事URL
```

※ タイトルはファイル内のh1見出しから自動取得します。
※ 有料記事リンクは固定のため入力不要です。
※ note記事URLが未指定の場合、Step 9のスレッドでは仮URL `https://note.com/chackwill/n/abc` を使用します（公開後に手動で差し替え）。

## 処理手順

### Step 1: 情報の読み込み

1. 指定された記事ファイルを読み込む
2. `SNS運用/note/` 内の過去記事を確認し、フォーマット・文体の基準を把握する
3. `_ kiwami/my-clone/` 内の人格データを参照し、文体・トーンを反映させる

### Step 2: 記事の品質チェック

以下の観点で問題を調査する：

- **重複チェック**: 過去記事と内容が重複している章・セクションがないか確認。重複があれば該当セクションを削除し、代わりとなる新しいセクションを生成する
- **事実確認**: 情報の正確性に疑義がないかチェック
- **文章品質**: 冗長な表現、論理の飛躍、読みにくい箇所の改善
- **web制作者であることは記述しない**

### Step 3: SEO最適化 + 内容充実化 + W16学習の反映

参照: [[SNS運用/analytics/W17戦略メモ.md]] と [[SNS運用/Threads運用.md]] の7つの型

1. **タイトルのSEO最適化 + 強フック化**
   - 検索されやすいキーワードを含める
   - noteのタイトル文字数制限（全角40文字程度）を意識する
   - **必須**: 以下7つの型のいずれかに当てはめる
     - 強い冒頭（命令形＋否定）/ 具体的な数字 / 常識を覆す / 対比 / 読者の痛み / 今すぐ行動 / 損失回避
   - **RT誘発**: 可能なら「上司」「同僚」「会社」などの関係性キーワードを検討
   - 数字・具体性・感情を入れてクリック率を高める

2. **本文のSEO最適化 + 1秒フック**
   - 見出し（h2/h3）にキーワードを自然に含める
   - **導入文の冒頭1文**は強い動詞で始める（「今日変わった」「〇〇が死んだ」「あなたの〇〇が」など）
   - 「昨日の続き」「〇日目」などの前提依存を削除（第一話化）
   - 内部リンクや関連情報の追加で記事を充実させる

3. **内容の充実化 + 保存誘発**
   - 追記できる情報があれば追加して記事の価値を高める
   - 各ニュースの実務観点での感想・分析を強化
   - 読者が「次に何をすべきか」が明確になるアクションを提示
   - **必須**: 各章に1つ以上のチェックリスト or 番号付きリストを入れる
   - **必須**: プロンプト例は `コピペ用` コードブロックで明記
   - **推奨**: 各章の末尾に「この章を保存して、〇〇する時に使って」の保存誘導を1文入れる

4. **参考元記事のURL明記**
   - リライト時に参考にした元記事・情報源のURLを各セクションに明記する

### Step 4: フォーマット整備

過去記事（`SNS運用/note/` 内）のフォーマットに合わせる：

- **冒頭の導入文**（タイトル直下のSEO用リード文と「みなさん、おはようございます！」から始まる挨拶導入文は内容が重複しやすいため、**1つの導入ブロックに統合する**。構成：「みなさん、おはようございます！今日のAIニュースまとめです。」で始め、昨日との接続→今日のニュース概要→テーマの提示→「今日もN本でいきましょう！」で締める。タイトル直下に別途SEO概要文を置かない）
- **目次セクション**（「この記事で分かること」と「ニュース一覧」は統合して1つの目次にする。各ニュースの見出しに「何が分かるか」の補足を添え、SEOキーワードを自然に含める。「この記事で分かること」と「今日のAIニュースN本(目次)」を別々に置かない）
- 各ニュースの構成: 概要 → 実務観点での感想 → ポイントまとめ → 情報源リンク
- セクション間の `---` 区切り
- トレンドまとめ・3レイヤー分析（該当する場合）

#### 🔴 太字（`**...**`）の使い方ルール（必須・厳守）

**目的**: 読者の目を「キーワード」に誘導するために太字を使う。文全体や長い節を太字にすると強調が拡散して逆に読みづらくなる。

**ルール**:

1. **太字は20字以内のキーワード／短いフレーズ限定**
   - ✅ OK: `**評価軸の切り替わり**` / `**ROIで何人分浮かせた**` / `**ゼロ回答**`
   - ❌ NG: `**評価軸が「使ってます」から「ROIで何人分浮かせた」に切り替わった**`（長すぎ。文を太字にしている）
2. **「実務で効く3つの視点」の番号付き太字は最大15字**
   - ✅ OK: `**1つ目: A4 1枚で見極め3行を書く**` / `**2つ目: 業者リストに1社追加**`
   - ❌ NG: `**1つ目、AI物理インフラ投資の見極め3行をA4 1枚で書く**`（説明文を太字にしている）
   - フォーマットは `**Nつ目: <短いキーワード>**`。詳細は太字の外で書く
3. **「💾 この章を保存して、〜前夜」の囲み内も短く**
   - ✅ OK: `**業者一覧を30分で振られた前夜**`
   - ❌ NG: `**GW明けに上司から「AI関連の取引先一覧、30分で出して」と振られた前夜**`
4. **段落中の強調は「キーワード抽出」で行う**
   - ✅ OK: 「核心は**評価軸の切り替わり**。「使ってます」から「ROIで何人分浮かせた」へ──」
   - ❌ NG: 「核心は**評価軸が「使ってます」から「ROIで何人分浮かせた」に切り替わった**こと」
5. **例外的に20字超を許容する箇所**:
   - 5本紹介リストの番号付き見出し（章タイトルそのもの）
   - トレンドまとめの「物理＋ROI＋規制＋陣取り」のような**列挙キーワード**
   - 末尾の固定CTA（「スキ・フォローで応援してもらえると…」など）
6. **チェックリスト先頭の太字は10〜15字以内のラベル**
   - ✅ OK: `- [ ] **3社並行見積テンプレ**(対象機能／現単価／…)`
   - ❌ NG: `- [ ] **クラウド3社並行見積テンプレを四半期1回ルーチン化**(対象機能…)`

**自己チェック**: 記事を書き終えたら、`grep -oE '\*\*[^*]+\*\*'` で全太字を抽出し、**25字以上の太字が5個以上あれば書き直し**。意図的な例外（列挙・CTA）以外は20字以内に収める。

### Step 5: 有料記事リンクの設置

記事の最後に以下の有料記事へのリンクセクションを設置する：

### Step 6: タグ選定 + 保存

1. 記事の内容に最適なタグを **5つ** 選定する
   - noteで検索されやすいタグを優先
   - 例: `#AIニュース` `#生成AI` `#ChatGPT` `#AI活用` `#業務効率化`
2. 記事末尾に `## 📌 タグ` セクションとしてタグを追記する
3. 元のファイルに **上書き保存** する

> ⏭️ **重要リマインド**: Step 6 完了時点では **絶対に最終応答を返さないこと**。
> ここはまだ1/4の進捗です。続けて Step 7（サムネイル文言生成）→ Step 8（X Articles版）→ Step 9（Xスレッド版）→ Step 10（検証と完了報告）に進んでください。

### Step 7: サムネイル文言の生成（メイン＋各見出し）

note記事のメインサムネイルと各項目見出しサムネイルに使う「メインタイトル」「サブタイトル」を選定し、別ファイルとして保存する。

#### 生成するもの

1. **メインサムネイル**（記事トップに1枚）
   - **メインタイトル**: 12〜20字。記事全体のフックを凝縮した1行
   - **サブタイトル**: 20〜35字。メインを補完する具体的なベネフィット・数値・所要時間

2. **各見出しサムネイル**（各ニュース／章ごとに1組）
   - 記事内の `## 数字. 見出し` にマッチする章すべてが対象（通常は5章）
   - それぞれに **メインタイトル（12〜20字）** と **サブタイトル（20〜35字）** を生成

#### 選定ルール（7つの型＋関係性キーワード＋強調語）

参照: [[SNS運用/Threads運用.md]] の7つの型

- メインタイトルは以下の7つの型のいずれかに当てはめる：
  - ① 強い冒頭（命令形＋否定） / ② 具体的な数字 / ③ 常識を覆す / ④ 対比 / ⑤ 読者の痛み / ⑥ 今すぐ行動 / ⑦ 損失回避
- **RT誘発**: 可能なら「上司」「同僚」「会社」などの関係性キーワードを仕込む
- サブタイトルは「何が手に入るか」「何がわかるか」を明示し、メインと意味が重複しないこと
- 章の具体内容（数字・固有名詞・読者の痛み）から引き出す。**コピペや抽象的表現はNG**
- **強調語（ゴールド表示部分）**: メインタイトルの中で、画像生成時にゴールド色で強調する語を **1〜5字** で必ず1つ選定する
  - 選定優先順位: ①`"..."` や `「...」` で囲まれた語 → ②タイトルの結論部・ベネフィット語 → ③数字・固有名詞
  - 「忘れない」「8月までに」のような短い句もOK。**メインタイトルの完全部分文字列**であること（マッチしない語は禁止）

#### 保存先

- `SNS運用/title/title-YYYYMMDD.md`（日付は記事の日付）
- 事前にディレクトリ不存在なら `mkdir -p "SNS運用/title"` で作成してから保存

ファイル末尾に **5見出し一括生成用の統合ブロック** も配置する（背景画像5枚を一度にChatGPTへ流す用途）。

### Step 8: X Articles版の生成

note記事をXの「Articles」機能向けに変換し、別ファイルとして保存する。
参考: `SNS運用/post/Xへ記事転載.md` の戦略に準拠。

#### 変換ルール

1. **冒頭フック（タイトル直下）**
   - noteの「みなさん、おはようございます！」挨拶を削除
   - 代わりに**X向けの強いフック**（1〜3文）で始める。ニュースの中で最もインパクトのある事実を冒頭に置き、読者の手を止めさせる
   - **W16学習**: 可能なら1文目に「上司／同僚／会社」の関係性キーワードを仕込む（RT誘発）
   - 例: 「AI業界の「王座」が今日入れ替わった。Claude Opus 4.7がGPT-5-4 Proをぶち抜いた衝撃を5本で解説する。」
   - 例（関係性トリガー強め）: 「あなたの上司が来月"Claude入れろ"と言い出す理由が、今日のニュースで確定した。」

2. **note固有ブロックの削除**
   - `> 📢 AIニュースを追うだけで満足してませんか？` の宣伝ブロックを削除
   - `## 📢 「ニュースを読んで終わり」にしたくない人へ` の有料記事CTAセクションを削除
   - `## 📌 タグ` セクションを削除

3. **本文の調整**
   - markdownの太字（`**`）を削除し、プレーンテキストにする（XのArticlesエディタでは太字が反映されないため）
   - `### 実務観点での感想` → `### 実務で何が変わるか` に見出しを変更
   - `### ○○のポイントまとめ` → `**ポイント**` に簡略化
   - `🔗 **情報源:**` → `🔗 情報源:` に簡略化
   - 各セクションの引用ブロック（`> この章のポイント`）→ 太字の箇条書きに変換
   - 「です・ます」調を「だ・である」調に統一（Xの記事は断定調のほうがエンゲージメントが高い）

4. **末尾CTA（note誘導）**
   - 記事の最後に以下のnote誘導CTAを設置:
   ```
   📝 各ニュースの「自分の仕事にどう活かすか」をさらに深掘りした記事はnoteで毎日更新中。プロンプト集や実践テクニックも公開しています。

   👇 特に反響の大きかった記事はこちら
   🔗 [【PDF付】月80時間の残業→ゼロ。30個のプロンプトで人生が変わった話](https://note.com/chackwill/n/nd015195d2717)

   毎日のAIニュース＋実務で使えるテクニックを発信中。フォローで明日の記事もお届けします👇
   ```

5. **ハッシュタグ（末尾）**
   - X向けのハッシュタグを5つ選定し、記事末尾に1行で記載
   - 例: `#AIニュース #ClaudeOpus4 #OpenAI #生成AI #AI最新情報`

#### 保存先

- `SNS運用/post/article/X-YYYYMMDD.md`（日付は記事の日付）

### Step 9: Xスレッド版の自動生成（`thread`スキル連携）

Step 1〜8 完了後、続けて `thread` スキルの処理を実行する。
これにより、**1コマンドで「note記事リライト → サムネイル文言 → X記事転載版 → Xスレッド版」まで一気通貫**で生成される。

#### 入力データの組み立て

`thread` スキルに以下の3点を渡す（内部処理として実行）：

1. **記事ファイルパス**: Step 1〜6 で保存したリライト後のnote記事ファイルパス
2. **note記事タイトル**: リライト後のh1タイトルを自動取得
3. **note記事URL**:
   - 入力時に2行目でURLが指定されていれば、そのURLを使用

#### 11-2-ter. 本日タイムラインの固定スケジュール（🔴 絶対必須・2026-05-02 改訂）

| 時刻 | 種別 | アクション内容 | 元ファイル |
|---|---|---|---|
| **09:00** | 🐦 | **day{N} 投稿①（朝）** | `SNS運用/post/day{N}.md` `## 投稿①（朝）` |
| **09:15** | 🧵 | **Threads-day{N} 投稿①（朝）短縮版** | `SNS運用/threads/Threads-day{N}.md` `## 投稿①（朝）` |

2. ファイル本文に投下時刻の明示指定があればそれを優先（例: `> 投下: 12:00`）。明示なければ上表のデフォルト時刻を使う
3. day*.md / Threads-day*.md に `## 投稿③` 以降のセクションがある場合は、後述「B. 変動枠の埋め込みルール」の空きスロットに追加投下する（ルーチン枠を侵さない）

**🚫 批評型リプはスケジュールに含めない（2026-05-03 変更）:**

- 朝・昼・夜の批評型リプ枠（旧 09:30 / 13:00 / 19:30）は**自動生成しない**
- ユーザーが空き時間に随時 1〜数本対応する運用にシフト（GW中等の外出多い日でも詰まらない設計）
- 戦略メモ B項目にルーチンBとしての記載があっても、本日アクションプランには**起こさない**
- KPI記録（21:00）の集計対象には**入れない**（実行本数が日次でブレるため、週次で別集計）

戦略メモ由来の以下のような **日替わりタスク** は、ルーチン枠（A）の隙間に時系列順で詰め込む：

- Day Oneプロモ（X案・Threads案）

**空きスロット（埋め込み先候補）:**

14:00 / 14:30 / 15:00 / 15:30 / 16:00 / 16:30 / 17:00 / 17:30 / 18:00 / 18:30

#### 11-2-bis. 「note公開後の実URL差し替え」タスクの記述ルール（厳守）

10:00 のnote公開セクションに「公開後に実URLへ差し替える」タスクを書く際は、**差し替え対象を以下のホワイトリストから機械的に選ぶこと**。憶測でファイルを増やさない。

**✅ 差し替え対象に含めてよいファイル:**

1. **Thread版**: `SNS運用/post/thread/YYYYMMDD_note紹介スレッド.md`
   - 仮URL `https://note.com/chackwill/n/abc` を含むため必ず対象
2. **day*.md / Threads-day*.md** ※条件付き

**❌ 差し替え対象に含めてはいけないファイル:**

  - **「念のため確認」ですら書かない**（毎日不要なチェック工数が発生するため）

1. 対象候補ファイルを生成・特定した直後に、各候補ファイルを `Grep` で `note\.com/chackwill/n/abc` を検索
2. ヒットしたファイルだけを「実URL差し替え」リストに含める
3. Article版（`article/X-*.md`）は**Grep結果に関わらず常に除外**
4. 完了チェックリストの行数も実際の対象ファイル数と一致させる

#### 11-6. Apple Reminders への自動同期 ⭐ NEW 2026-05-01

`SNS運用/action-YYYYMMDD.md` の保存・検証完了直後に、`/sync-reminders` スキルの処理を**自動で呼び出す**。タイムライン早見表の各エントリを macOS Reminders の `Tsukapon` リストに登録し、iPhone・Apple Watch でも時刻通りに通知を受けられる状態を作る。

**実行コマンド（Bashツールで実行）:**

```bash
# 1. リスト存在確認
reminders show-lists | grep -q "^Tsukapon$" || reminders new-list "Tsukapon"

# 2. 既存の同タイトルを取得して重複防止
existing=$(reminders show Tsukapon --format json 2>/dev/null | python3 -c 'import json,sys; data=json.load(sys.stdin); [print(r["title"]) for r in data]' 2>/dev/null)

# 3. 早見表の各行を抽出して登録
#    タイトル形式: "HH:MM {アイコン} {内容}"
#    --due-date は "YYYY-MM-DD HH:MM" 形式
reminders add Tsukapon "HH:MM 🧵 内容" --due-date "YYYY-MM-DD HH:MM"
```

**スキップ条件:**

- 同タイトルが既に `Tsukapon` リストに存在する → スキップ（重複防止）
- 9時前の時刻枠（運用制約・CLAUDE.md「📅 タイムライン早見表」ルール準拠）→ スキップ
- `reminders` コマンドが見つからない（未インストール）→ 警告のみで続行

**完了報告（Step 10 の追加項目）:**

- 「⏰ Apple Reminders 同期: N件追加 / M件スキップ（重複）」を完了報告に1行入れる
- 失敗時は「⏰ Apple Reminders 同期失敗（理由: ○○）」と理由を1行報告

**禁止事項:**

- `Tsukapon` 以外のリストには絶対に書き込まない
- 既存リマインダーのタイトル変更や削除はしない（ユーザー操作を尊重）
- Step 11 全体の処理を Reminders 同期の失敗で止めない（best-effort）

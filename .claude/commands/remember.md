重要な気づき・決定・運用変更を、適切な階層のメモリファイルに振り分けて追記するスキル。Hermesエージェントの「多階層メモリ」発想をObsidian vault運用に持ち込んだもの。

## 用途

- セッション中に出てきた「これは記録しておくべき」な情報を、`_ memory/` の short / mid / long に正しく振り分ける
- 揮発性に応じて適切な層に置くことで、後から読み返したときにノイズ少なく辿れる
- 「あれどこに書いたっけ」状態を防ぐ：判断は `/remember` に集約

## 入力形式（4パターン対応）

**パターン A: 階層指定 + 本文**
```
/remember short Hermes Kanban v1.5.0からは思想だけ取り込む方針
/remember mid 2026 Q2はThreads様子見ステージ
/remember long リプ欄スパムは絶対やらない
```

**パターン B: 本文のみ（自動振り分け）**
```
/remember 2026-01改定でリプ欄インプは収益化対象外
```
→ 内容と時制から階層を推定してユーザーに確認してから書き込む。

**パターン C: エディタ選択 / ブラウザ選択**
- 引数が空で `<editor_selection>` または `<browser_selection>` がある場合、その内容を対象にする
- 出典（ファイルパス／URL）も合わせて記録

**パターン D: 引数なし（対話モード）**
```
/remember
```
→ 「何を覚えますか？」と聞き、続けて階層を提案する。

## 階層の判定ガイド

| 層 | レンジ | 特徴 | 例 |
|---|---|---|---|
| **short-term** | 1〜4週間 | 揮発性高・上書きOK | 今週のフォーカス、直近の試行、期限あり宿題 |
| **mid-term** | 3〜6ヶ月 | 運用方針・現役の戦略 | 四半期の運用方針、現役スキル一覧、定常的な前提 |
| **long-term** | 半年以上〜恒久 | 価値観・鉄則・人格 | 行動原則、文体の根、vault恒久構造、絶対禁止事項 |

**判定キーワード（迷ったときの目安）**:

- 「今週」「直近」「試行中」「期限」 → short-term
- 「今四半期」「現在の方針」「Q2」「現役の」 → mid-term
- 「絶対」「鉄則」「不変」「価値観」「always / never」 → long-term

## 処理手順（6ステップ）

### Step 1: 入力の確定

- 入力パターンを判定し、本文 `BODY` と階層 `TIER`（指定があれば）を確定
- パターンB（自動振り分け）の場合は、判定キーワードから候補階層を提案 → ユーザーに確認
- 出典情報（パス／URL）があれば記録

### Step 2: 対象ファイルの特定

| TIER | ファイル |
|---|---|
| short | `_ memory/short-term.md` |
| mid | `_ memory/mid-term.md` |
| long | `_ memory/long-term.md` |

ファイルが存在しない場合はテンプレ付きで新規作成（既存テンプレートに従う）。

### Step 3: 追記位置の決定

各ファイルには既にセクションが切られている。本文の性質に応じて適切な見出し配下に追記する：

- **short-term.md**:
  - 「今週のフォーカス」: 今週のタスク・進捗
  - 「直近の決定・気づき」: 日付つき発見・判断
  - 「試行中の運用変更」: 実験中の手順変更
  - 「期限・宿題」: 期限ありTODO
- **mid-term.md**:
  - 「現在の運用戦略」: 四半期方針
  - 「Claudian運用方針」: スキル運用ルール
  - 「現役のスキル」: スキル表更新
  - 「現在の運用前提」: 前提条件の変更
- **long-term.md**:
  - 「行動原則」: 不変の原則追加
  - 「文体・人格の根」: 文体ルール追加
  - 「vaultの恒久構造」: パス追加
  - 「触ってはいけないもの」: 禁則追加
  - 「価値観」: 哲学追加

セクションの判定が曖昧な場合はユーザーに確認する。

### Step 4: 追記の実行

- `Edit` ツールで該当セクションの末尾に1項目追加
- 日付つきが望ましい項目には `**YYYY-MM-DD**:` プレフィックスを付ける
- 出典がある場合は末尾に `（出典: [[path]]）` または `（出典: [タイトル](URL)）`
- frontmatter の `last_updated` を今日の日付に更新

### Step 5: 階層昇格・降格の検討

追記後、以下を一瞬チェック：

- short-term に1ヶ月以上居座っている項目がないか → mid昇格 or 削除を提案
- mid-term に半年以上同じ項目がないか → long昇格を提案
- long-term に「これもう古い」項目がないか → 降格 or 削除を提案

該当があれば「ついでに整理しますか？」と1行だけ提案。**勝手に動かさない**。

### Step 6: 完了報告

```
🧠 メモリに記録しました

📂 階層: {short / mid / long}
📄 ファイル: [[_ memory/{tier}-term.md]]
📍 セクション: {追加先のセクション名}

追記内容:
> {追加した本文を1行プレビュー}

{昇格/降格の提案があれば: 「💡 ついでに〜の整理どうする？」}
```

## 制約・安全運用

- **既存項目を書き換えない**（追記のみ）
- frontmatter `last_updated` 以外の既存メタデータは触らない
- 同じ内容が既に書かれていないか軽くチェック（重複追記の警告）
- 1度に複数項目を入れる場合は1つずつ確認しながら追記
- ユーザーが明示的に階層を指定したら自動振り分けは行わない（指定を尊重）
- `_ memory/` 配下は自動リンクルール（`🔗 関連コンテンツ`）の対象外（メモリ層は素のまま保つ）

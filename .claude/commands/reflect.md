最近の会話・作業を振り返り、繰り返し現れる作業パターンや改善ポイントを抽出して `Claudian-スキル候補.md` に追記するスキル。Hermesエージェントの「自己改善ループ」発想をObsidian vault運用に持ち込んだもの。

## 用途

- セッション末尾やまとまった作業の後に、Claudian自身が「今日繰り返した作業」を棚卸しする
- 同じパターンが複数回出てきたら、新スキル候補としてバックログに積む
- ユーザーが「最近こういう作業よくやってない？」と聞いたとき、定量的な答えを返せる土台にする

## 入力形式（3パターン対応）

**パターン A: 引数なし（デフォルト）**
```
/reflect
```
→ 直近のセッション（このチャット内のやり取り）を対象に、パターン検出 → 候補追記。

**パターン B: 範囲指定**
```
/reflect today
/reflect week
/reflect last-3-sessions
```
→ 指定範囲の作業を対象（`Daily Log/` や `_ kiwami/tools/daily-log/` のログを参照）。

**パターン C: 特定パターンを記録**
```
/reflect "毎回 archive/ に手動で移動している"
```
→ ユーザーが明示的に観察したパターンを直接候補に追加。

## 処理手順（7ステップ）

### Step 1: 対象範囲の確定

- 入力パターンを判定して `SCOPE` を確定
- パターンAの場合は「現在のセッションでアシスタントが行ったタスク一覧」を対象
- パターンBの場合は該当範囲のログファイルを読む

### Step 2: パターン抽出

以下の観点でタスクを分類・カウント：

1. **ファイル操作パターン**:
   - 同じディレクトリへの繰り返しWrite / Edit
   - 命名規則が共通（`day*.md`、`YYYY-MM-DD-*.md` など）
   - 同じ移動先（`SNS運用/archive/`、`調査/` など）への繰り返し操作
2. **会話パターン**:
   - ユーザーが似た形式の指示を繰り返している（「〜を整理して」「〜を移して」など）
   - 同じスキルを繰り返し呼んでいる（→ そのスキルの拡張余地あり）
   - スキルがない領域で手作業が3回以上発生している
3. **改善ポイント**:
   - 既存スキルの「あと一歩足りない」フィードバック
   - ユーザーが手で修正している部分（自動化候補）

### Step 3: 既存候補との突合

`Claudian-スキル候補.md` を読み、既に同じパターンが登録されていないか確認：

- 既存エントリと一致 → カウントを+1して `検出日時` を最新化
- 新規パターン → 新エントリとして追加

### Step 4: ステータス遷移

検出回数に応じてステータスを更新：

| カウント | ステータス |
|---|---|
| 1〜2 | 🟡 観察中 |
| 3以上 | 🟢 候補確定 |

3回到達したエントリは、🟢セクションへ移動する。

### Step 5: 既存スキルへの再帰チェック

抽出パターンが**既存スキルの拡張**で済まないか確認：

- 例: 「`/deep-dive` を呼んだ後、毎回出力に同じ整形をしている」
  → 新スキルではなく、`/deep-dive` の出力テンプレ更新を提案
- 既存スキル拡張で済むなら、候補ではなく**既存スキルへの改善メモ**として `_ memory/short-term.md` に書く

### Step 6: 追記の実行

`Claudian-スキル候補.md` を `Edit` で更新：

- 新規エントリは適切なセクション（🟡 / 🟢）に追加
- 既存エントリはカウント・日付を更新
- frontmatter `last_updated` を今日の日付に

### Step 7: レポート提示

```
🔍 振り返り完了

📊 検出範囲: {SCOPE}
🟡 新規 観察中: N件
🟢 候補確定（3回到達）: N件
🔄 既存エントリ更新: N件

ハイライト:
1. [{ステータス}] {パターン名} — 検出 {N}回
2. [{ステータス}] {パターン名} — 検出 {N}回
3. ...

💡 候補確定したものから実装するならどれにする？
（実装決定するなら `Claudian-スキル候補.md` で 🔵 に動かして、その後 `.claude/commands/` に新スキルを作ります）
```

## パターン検出のヒント（Claudian自身向けの認知ガイド）

**よくある繰り返し作業の例**:

- 「今日のX投稿を `archive/` に移動」← `/archive` で解決済み
- 「外部記事を読んで `調査/` にまとめる」← `/deep-dive` で解決済み
- 「複数のClippingsを横断して関連ノートを作る」← 候補になりうる
- 「特定タグの一覧を集計する」← 候補になりうる
- 「特定フォルダ内の `tags:` を統一する」← 候補になりうる
- 「Daily Log から週次サマリを作る」← 候補になりうる
- 「`SNS運用/note/` の記事を画像生成つきで公開準備」← 候補になりうる

## 制約・安全運用

- **勝手にスキルを実装しない**。`/reflect` は候補追加までで止める
- 候補ファイルの既存エントリの**実装案ドラフト**を勝手に書き換えない（追記のみ）
- ⚪ 見送り に入っているパターンは再提案しない（同じ判断ループを避ける）
- 1回の実行で大量（10件以上）の新規候補が出たら、ユーザーに「絞り込みする？」を確認
- 検出は**観察**であって**評価**ではない。ユーザーのワークフローを否定しない

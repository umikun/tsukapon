## 1. 利用目的

サイボウズ Office のメッセージ機能にはプッシュ通知やモバイル通知の仕組みがなく（契約プランによる）、未読に気づくにはブラウザで都度確認する必要がある。
個人宛のメッセージを5分毎に確認し、Chatworkで受信することで業務指示確認のロスタイムを短くしたい。

## 2. Chatworkを使う理由（メール等では代替できない理由）

社内のSEチーム間で日常的にChatworkを業務連絡に使用しており、一元管理することで通知の確認先を増やさずに済む。

## 3. 連携内容（送信/取得するデータ、投稿先ルーム、頻度）

| 項目 | 内容 |
|------|------|
| データ | サイボウズに送られてくる個人宛のメッセージの送信者名・メッセージタイトルのみ。本文内容は送信しない |
| 投稿先ルーム | マイチャット |
| 頻度 | 5分毎（設定変更可） |

## 4. 利用ツール・実装方式

添付の「サイボウズOffice 未読メッセージ → Chatwork 通知の概要（案）.md」ファイルに沿って実装予定。

## 8. 代替手段と比較したうえでの採用理由

検討した代替手段：
1. 手動での定期確認（確認漏れ・タイムラグが課題）
2. サイボウズのメール通知機能 ⇒ 契約プランにより、利用不可
3. Chatwork API連携（通知の即時性と一元管理の両立が可能）

業務関連の通知一元管理という観点から、現在利用しているChatworkを使用するのが一番現実的と判断した。

## 9. セキュリティ対策

- プログラムの実装ディレクトリはローカル環境でのみ使用し、Git管理・共有等は行わない。
- 未読画面をスクレイピングで監視する手法には情報漏洩のリスクが伴うが、社内PCから当該画面にアクセスする運用を前提としているため、社外への情報流出は想定されない。

## 10. Chatwork APIの利用量について

- レート制限：300リクエスト / 5分（ユーザー単位）
- 利用量見込み：1日最大288回（= 24時間 × 60分 ÷ 5分）新着がある場合のみPOSTするため、実際にはこれ以下に収まる。

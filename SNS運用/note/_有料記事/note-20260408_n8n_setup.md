# 月額約4,600円のZapierをやめて無料の「n8n」に乗り換えたら、完成後にも地獄が待ってた話

**先に結論。できた。でも3日かかった。そして「完成」は終わりじゃなかった。**

僕はしがないポンコツWeb制作者なんですけど、プログラムの知識はほぼない。WordPressのカスタマイズくらいならなんとかなるけど、APIとかサーバーとかになると毎回ググりまくり💦

そんな人間が「Claude Codeに聞きながらやれば2〜3時間でいけるっしょ」って軽い気持ちで無料ツールに手を出した結果、地獄を見ました。

同じことやろうとしてる人、この記事読んでからにしてください。マジで。

ちなみに今回の作業は全部Macでやってます。Windowsでも同じことはできるはずだけど、動作確認はしてないんで、そこはご了承ください🙇‍♂️

---

## そもそも何がしたかったのか

作業時間の管理に「Clockify」っていう無料ツールを使ってて、そこに登録した作業内容をZapier経由でGoogleカレンダーに自動で飛ばしてた。

ちなみにClockifyの前は「Toggl」を使ってたんだけど、2025年6月の仕様変更で無料プランでは制限がかかるようになってしょうがなく移行した。無料ツールの宿命よね。

で、Zapierも無料プランは制限がキツい。有料だと月額29.99ドル（約4,600円）かかる。

「ローカルで動く無料のやつないかな」って探したらn8nっていうオープンソースのツールを見つけた。

しかもこれ、ZapierだけではできなかったDay One（日記アプリ）への同時記録もできそう。Day Oneへの登録は今まで「IFTTT」を使ってたんだけど、n8nに一本化できればIFTTTも無料プランに落とせる。

```
Clockify（作業時間を記録）
 ↓ 自動で
Google Calendar（カレンダーに表示）
 ↓ 同時に
Day One（日記にも作業ログを残す）
```

「いけるんじゃね？」

この判断が間違いの始まりだった。

## Day 1：インストールから7時間の格闘

### インストールはあっさり終わった

npmかDockerか迷ったけど、負荷が軽そうなnpmにした。

```bash
npm install n8n -g
```

起動も `n8n start` で一発。ブラウザで http://localhost:5678 を開いたら管理画面が出た。

ここまでは順調順調。ここまでは。

### Google連携で30分ハマる

n8nからGoogleカレンダーに書き込むにはOAuth認証が必要で、Google Cloud ConsoleでClient IDを作らないといけない。

ここまではなんとかなった。問題はその先。

「承認済みのリダイレクトURI」の設定場所がわからない。

Google Cloud Consoleの画面をウロウロして、結局AIに聞いてやっと見つけた。設定すべき値はこれ：

```
http://localhost:5678/rest/oauth2-credential/callback
```

たったこれだけのことに30分💦

### 動いた！...と思ったら大量の予定が爆誕

ワークフローを組んで実行したら、過去のエントリが全部まとめてGoogleカレンダーに登録された。しかも実行するたびどんどん増えていく💦

カレンダーが予定で埋め尽くされてる。重複どころの話じゃない。

「最新1件だけ取得」「処理済みIDを記録してスキップ」っていう仕組みを入れて、なんとか解決。

あと作業時間が「PT2H」って表示されて「は？」ってなった。ISO 8601形式とかいうやつらしい。「2:00」って表示したいだけなのに。

結局、「PT」を外して「H」を「:」に変換、分がなければ「:00」を付ける、みたいな変換処理をJavaScriptで書くハメになった。たかが時間の表示にコードを書く日が来るとは思わなかった。

### Day One連携がこの日最大の地獄

「せっかくだからDay Oneにも記録したい」って欲を出した。これが間違いだった。

Day Oneにはターミナルから日記を書き込めるCLIツールがある。まずインストール。

```bash
sudo bash /Applications/Day\ One.app/Contents/Resources/install_cli.sh
```

n8nにJavaScriptコードのノードを追加して、このCLIを叩くコードを書いた。

エラー。

`child_process`ってモジュールが使えないらしい。n8nはセキュリティの都合でNode.jsの機能を制限してるんだけど、そんなのしがないポンコツWeb制作者は知らないって。

環境変数を付けて再起動しないとダメだった：

```bash
NODE_FUNCTION_ALLOW_BUILTIN=child_process,fs n8n start
```

これでやっと動いた。

...と思ったらDay Oneに登録された内容が「【：】2:00」だけ。

クライアント名もプロジェクト名も空っぽ。データの参照先が間違ってた。

修正して、ようやくちゃんとした形で登録できた：

```
【clientName：projectName】description
> Start：April 3, 2026 at 12:00 PM
> End：April 3, 2026 at 2:00 PM
> 作業時間：2.0h
```

「2〜3時間で終わるだろ」と思って始めたのに7時間かかった。

## Day 2：翌朝、何も動いてない

意気揚々とClockifyに作業を登録。

Googleカレンダーを確認。

何も起きない。

Day Oneも確認。何も登録されてない。

は？

### 「動いてません」を何回言ったか

手動テストだと動く。でも自動では動かない。

- ログを確認 → エラーなし
- ワークフロー再起動 → 変化なし
- n8nごと再起動 → 変化なし
- デバッグモードで起動 → 手がかりなし

何回やっても動かない。この時点でかなり心折れかけてた。

### 原因①：Publishしてなかった

...恥ずかしすぎる。

n8nのワークフローって、作っただけじゃ自動実行されない。「Publish」ボタンを押して初めて本番稼働する。

手動テストでは動くから盲点だった。

Publishして「よし！」と思ったら...

まだ動かない。

### 原因②：Clockify Triggerが根本的にダメだった

n8nにはClockify専用のTriggerノードがあるんだけど、これが新しいエントリを全然拾ってくれない。

内部データをリセットしたり、デバッグログを追ったり、散々やった。ダメ。何してもダメ。（やったのはClaude codeですが）

直接入力（タイマーじゃなく手動で時間を記録する方式）との相性問題なのか、ノードのバグなのか。最後まで原因不明。

### 方式を丸ごと変えた

正直「作り直すの面倒」って思ったけど、動かないものにしがみついても仕方ない。

Clockify Triggerを捨てて、Schedule Trigger + HTTP Request方式に切り替えた。

```
旧：Clockify Triggerが変更を検知 → 処理実行
新：毎分スケジュール実行 → Clockify APIで最新データを取得 → 新しければ処理
```

エレガントじゃない。でも確実に動く。もう「動く」が正義。

### さらにワークスペースIDが間違ってた

新方式でClockify APIを叩いたら、データが取れない。

調べたら、APIのURLに指定してたIDが違ってた。ワークスペースIDの欄にユーザーIDを入れてた。

```
# 間違い（ユーザーIDだった）
.../workspaces/684043134d7e18788e95a3d4/user/...

# 正解（ワークスペースID）
.../workspaces/684043134d7e18788e95a3d5/user/...
```

末尾が d4 と d5 の1文字違い。こんなの気づかないって💦

修正したら、ようやく全部つながった。

この瞬間は声出た。

## Day 3：仕上げ

動くのが確認できたので、あとは仕上げだけ。

- **重複防止：** 処理済みエントリのIDをファイルに保存して、同じIDならスキップ。毎分動くのでこれがないと同じデータが何度も登録される。
- **自動起動：** 毎回ターミナルで起動するのは面倒なんで、macOSのLaunchAgentで設定。Macを起動するだけでバックグラウンドでn8nが動く。

## 完成したワークフロー（Day 3時点）

```
Schedule Trigger（3分間隔）
 ↓
HTTP Request（Clockify APIで最新1件を取得）
 ↓
Filter New（前回と同じIDならスキップ）
 ↓
├→ Google Calendar（イベント作成）
└→ Day One（CLIでジャーナル登録）
```

**Googleカレンダー側**

- タイトル：【clientName：projectName】description
- 場所：Location
- 説明：作業時間（例：2:00）

**Day One側には下記も追加**

- ジャーナル：Job
- タグ：clientName

```
【clientName：projectName】description
> Start：April 3, 2026 at 12:00 PM
> End：April 3, 2026 at 2:00 PM
> 作業時間：2.0h
```

これで完成。やっと終わった。

...と思ってた。

## Day 4〜5：「完成」したはずのワークフローが壊れてた

数日間、普通に使ってた。Clockifyに登録すればカレンダーにもDay Oneにも反映される。順調順調。

ところがふと気づいた。

**「あれ、昨日の作業、カレンダーに入ってない」**

1日に3件登録したうちの2件が消えてる。Day Oneも同じ。最後に登録したやつだけ残ってて、途中のが抜けてる。

最初は「たまたまかな」って思ったんだけど、何日か見てるとパターンがあった。**短い間隔で複数件登録すると、途中のやつが消える。**

### 原因調査：AIに丸投げ

正直、自力で原因を探せる気がしなかった。ログを見ても全部「success」って出てるし。

Claude Codeにn8nのデータベースを直接調べてもらった。SQLiteのファイルを読んで、1,000件以上の実行履歴を分析。

出てきた原因が3つ。しかも全部自分が作り込んだバグだった。

### 原因①（最大の問題）：最新1件しか見てなかった

Clockify APIから `page-size=1` で最新1件しか取ってなかった。

これ、何がマズいかっていうと——

```
09:00 ポーリング → エントリA検出 → 処理OK
09:01 エントリBを登録
09:02 エントリCを登録
09:03 ポーリング → エントリC（最新）を検出 → 処理OK
→ エントリBは永久に取りこぼし
```

**3分間隔で最新1件だけ見てるから、間に挟まったやつが消える。**

「最新1件取れば十分でしょ」って思ってたけど、全然十分じゃなかった。

### 原因②：処理が失敗してもIDだけ更新してた

重複防止のために「処理済みID」をファイルに保存する仕組みを入れてたんだけど、このID、**Google Calendar登録やDay One登録の「前」に保存**してた。

つまりこういうこと：

```
Filter New「新しいエントリだ！IDを保存っと」
 ↓
Google Calendar「...エラーで登録できませんでした」
 ↓
次のポーリング「このID、処理済みだからスキップね」
 → 永久にロスト
```

「処理済み」って言いながら処理できてないっていう。笑えない。

### 原因③：APIタイムアウトをリトライしてなかった

Clockify APIへの接続がたまにタイムアウトしてた。1,176回中19回。

少ないっちゃ少ないけど、Retry on Fail（自動リトライ）を設定してなかったんで、タイムアウトしたらそのまま失敗。その間に登録したエントリはスルーされる。

### 修正作業：さらに地獄を見る

原因がわかったんで修正。これも全部Claude Codeにやってもらった。

やったこと：

- **page-sizeを`1`→`10`に変更**（複数エントリの取りこぼし防止）
- **Filter Newのロジックを書き直し**（複数の新規エントリに対応）
- **ID保存のタイミングを修正**
- **Retry on Failを有効化**（タイムアウト時に3回まで自動リトライ）
- **Execute Onceの解除**（複数件を全件処理するため）

...で、一発で直ったかっていうと、全然。

**修正→テスト→別のエラー→修正→テスト→また別のエラー** の無限ループ。

n8nのコードエディタに貼り付けたら `?.`（オプショナルチェーン）が `? .` にスペース入りで変換されてSyntaxError。

Day OneのコードがGoogle Calendarのデータを参照しちゃって「そんなプロパティないよ」エラー。

処理済みIDのファイルにClockifyのIDじゃなくて**Google CalendarのイベントID**が書き込まれて、重複登録が大復活。

もう何回「Save → Publish」したかわからない。

### 最終的に直った

結局、修正に丸2日かかった。

最終的なワークフロー：

```
Schedule Trigger（3分間隔で実行）
 ↓
HTTP Request（Clockify APIで最新10件を取得、リトライ3回）
 ↓
Filter New（複数エントリ対応、実行中タイマー除外）
 ↓
Create an event（Google Calendar、全件処理）
 ↓
Code in JavaScript（Day One、全件処理）
```

Clockifyに3件まとめて登録しても、3件ともGoogle CalendarとDay Oneに反映される。重複もない。

**ここでやっと本当の「完成」。**

## 僕がハマったポイント10個

**構築フェーズ（Day 1〜3）**

1. Google OAuthのリダイレクトURI設定場所がわからない（30分ロス）
2. 過去エントリが全部登録されてカレンダーが大惨事（1時間ロス）
3. n8nのセキュリティ制限でDay One CLIが動かない（1時間ロス）
4. データマッピングミスでDay Oneに空データが登録（30分ロス）
5. ワークフローのPublishを忘れて半日「動かない」と騒ぐ（2時間ロス）
6. Clockify Triggerが根本的に動かず方式変更（3時間ロス）
7. ワークスペースIDとユーザーIDを取り違え（30分ロス）

**運用フェーズ（Day 4〜5）**

8. page-size=1で複数エントリの取りこぼし（調査1時間 + 修正2時間）
9. 処理前にIDを保存→失敗したら永久ロスト（修正1時間）
10. 修正のたびに別のバグが連鎖発生（合計3時間）

**合計：約15時間。** 構築8.5時間 + 改修6.5時間。もはや笑うしかない。

## Zapierからn8nに乗り換えてどうだったか

### 良かったこと

- 月額約4,600円がゼロになった。これがデカい
- IFTTTのDay One連携も不要になったんで、IFTTTも無料プランに落とせた
- データがローカル完結。外部サーバーを経由しない
- JavaScriptでなんでも書ける。Day One連携もこれで実現できた
- Zapierはwebサイトでログイン確認しないと動作しないことがたまにあったから毎回面倒だった
- Zapierじゃできなかった連携が追加できた

### キツかったこと

- Zapierなら30分で終わることに**5日かかった**
- 「完成」してからが本当の地獄だった
- ローカルで動くのでMacが落ちたら止まる（Macでしか登録しないんで問題なさそうだけど）
- トラブルは全部自力解決。ドキュメントも英語ばっかり
- エラーメッセージが不親切で原因特定がしんどい
- AI使わないと絶対できてなかった

**向いてる人：** 月額を削りたい人、データを外に出したくない人、トラブルを「ネタ」にできる人

**向いてない人：** 設定して即使いたい人、心がすぐ折れる人

## 正直に言う

途中で「Zapierの課金続けたほうが楽だったんじゃ...」って5回くらい思った。

特に運用フェーズの不具合は堪えた。構築時のトラブルは「まあ最初だし」って割り切れたけど、「動いてたはずのやつが実は壊れてた」っていうのは精神的にくる。

でも最終的に直った瞬間、3件まとめて登録して3件ともカレンダーとDay Oneに反映されたときは、ちょっと泣きそうになった。大げさじゃなく。

**「動いてるつもり」と「本当に動いてる」は違う。** 今回の一番の教訓はこれかもしれない。

月額もゼロになったし、Day One連携も手に入った。取りこぼしもなくなった。結果的にはやってよかった。

n8nが便利なのは間違いない。今回はClockifyの連携だけだったけど、これ1つで5,000以上のアプリと繋がれるらしい。Gmailの添付ファイルを自動でGoogle Driveに保存とか、Slackに定期レポートを投げるとか、やれることは山ほどある。しばらくはこいつでいろいろ試してみたい。

ただ、同じことやろうとしてる人に言いたいことが2つある。

**1つ目：「簡単にできる」って書いてあるブログ記事を信じるな。**

僕みたいなポンコツは、書いてある通りにやっても書いてある通りにならないから。

**2つ目：「動いた」で安心するな。**

数日間は注意して見てたほうがいい。「たまに取りこぼす」とか「特定の条件で壊れる」みたいなバグは、すぐには気づけないんで。

それでもやるなら、この記事を読んでおいてほしい。少なくとも「page-sizeは1にするな」と「ワークフローのPublishを忘れるな」、この2つだけは持ち帰れるはずなんで。

## 1. OpenAI、ChatGPT標準モデルを「GPT-5.5 Instant」に更新

### 要約

OpenAIは2026年5月5日、ChatGPTの標準モデルを **GPT-5.5 Instant** に切り替えました。GPT-5.3 Instantを置き換える更新で、**より正確で簡潔な回答**、過去チャット・ファイル・接続済みGmailを使った個人化、回答に使われた記憶情報を確認・削除できる **memory sources** が特徴です。APIでは `chat-latest` として提供され、日常利用の基盤が一段更新されます。

### 重要ポイント

- 全ユーザー向けの**標準モデル更新**で影響範囲が広い
- 医療・法律・金融など**高精度が必要な領域**で事実性改善を打ち出した
- **個人化の強化と同時に、記憶情報の透明性・削除権**が重要テーマに
- 有料ユーザー向けに **GPT-5.3 Instantは3カ月だけ選択可能**

### 実務で効く3つの視点

**1つ目: memory sourcesを開ける**
何を覚えさせて、何を忘れさせるか。**回答の根拠を1クリックで遡れる**ようになったので、機密情報・顧客名・社内固有名詞は**最低週1で監査**する。これだけでGDPR/個人情報保護法のグレーが半分減ります。

**2つ目: 切替を全社共有しない**
標準モデル更新は「いつの間にか変わってる」典型例。**経理・法務など定型回答を期待する部署**には、3カ月猶予のGPT-5.3 Instant継続をルール化しておくと、検証中の誤回答事故が減る。

**3つ目: APIは "chat-latest" で固定する誘惑を捨てる**
chat-latest は最新モデルに自動追従するため、**プロダクション利用ではバージョン固定**（`gpt-5.5-instant`）が無難。検証用環境だけ chat-latest で先行確認する2系統運用が安全です。

- [ ] **memory sources確認** — 自分のChatGPTで開いて、消すべき記憶3つピックアップ
- [ ] **3カ月猶予の通知文** — 高精度回答が必要な部署向けに、5.3 Instantが選べる旨を1段落で社内Wikiに書く
- [ ] **API切替判定** — 自社で `chat-latest` を使っているサービスがあれば、バージョン固定への移行可否を1人で判断
- [ ] **"記憶削除権" 文言追加** — 顧客向けFAQに「ChatGPTで保管された情報の削除依頼は○○へ」を追加するか議論

## 2. 米NIST/CAISI、Google・Microsoft・xAIの未公開AIモデルを事前評価へ

### 要約

米商務省NISTの **Center for AI Standards and Innovation (CAISI)** は、Google DeepMind、Microsoft、xAIと新たな協定を結び、**公開前のフロンティアAIモデルを評価**すると発表しました。国家安全保障上の能力・リスク、サイバー、バイオ、化学兵器関連の懸念を測定する枠組みで、OpenAIやAnthropicとの既存協力も再調整。**米国のAI監査体制が一気に広がります**。

### 重要ポイント

- 未公開モデルの政府評価が **OpenAI/Anthropicから大手3社へ拡大**
- CAISIはすでに **40件以上の評価を完了** していると説明
- **安全装置を弱めたモデル**も評価対象にし、実際の悪用リスクを測る
- 「規制で止める」より **「公開前に測る」方向**へ米国AI政策が動いている

### 実務で効く3つの視点

**1つ目: 評価軸の切り替わり**
「使ってます」から「**事前評価を通った**かどうか」へ調達の判断基準が変わる。来期、上司が「うちの使ってるAI、CAISI通ってる？」と聞いてくる可能性が出てきました。1社単独依存はそれだけでリスクです。

**2つ目: A4 1枚で見極め3行**
- 使っているモデル名とバージョン
- そのモデルの評価協定の有無（CAISI/AISI 等）
- 評価未通過時の代替モデル候補2つ

これを書けるかが**AI調達担当の最低ライン**になりました。

**3つ目: マルチベンダー化への布石**
CAISIが3社追加で「単独依存は終わり」のシグナルを出した。**LiteLLM や OpenRouter 等のラッパー**を1機能だけでも先に検証しておくと、「Azure一択」「Anthropic一択」の急な見直しに耐えられます。

## 3. Google、Gemini API File SearchをマルチモーダルRAG対応に拡張

### 要約

Googleは2026年5月5日、Gemini APIの **File Searchを拡張** し、画像とテキストを横断して検索できるマルチモーダル対応、カスタムメタデータによる絞り込み、PDFなどの **ページ単位引用** を追加しました。Gemini Embedding 2を使い、社内文書・画像資産・科学データを扱うRAGアプリの**検証性と実用性**を高める更新です。AI回答の根拠を業務で追いやすくなり、**監査対応にも効きます**。

### 重要ポイント

- RAGが **「テキスト検索」から「画像込みの業務記憶」** へ進む
- **ページ番号付き引用** により、AI回答の根拠確認がしやすくなる
- メタデータ絞り込みで、**部署・版・状態別の社内検索**に使いやすい
- エージェントに渡す文脈を圧縮し、**実務アプリの精度向上**につながる

### 実務で効く3つの視点

**1つ目: RAGの主戦場が画像へ**
製造業・医療・建築のように **「図面・写真・スキャンPDF」が業務の中心**な業界では、ここから本番。「テキストRAGは試した、写真RAGは未着手」の状態は半年で詰みます。

**2つ目: ページ単位引用＝監査の通過条件**
ニュース②のCAISI評価と直結。**「どのページから引いたか」が出ない RAGは、来期から監査で詰まる**可能性。File Searchの引用精度は実装前に必ず触っておく。

**3つ目: Embeddingモデルは半年で陳腐化する前提**
Gemini Embedding 2 で過去の埋め込みを**全件再計算する判断**が、来週どこかで降ってきます。再計算コストとベクトルDBの容量を**先に見積もっておく**だけで、慌てません。

### この章を保存して、社内RAGアプリの検証会で「画像対応は？」と聞かれた朝に

- [ ] **File Search ページ単位引用テスト** — 自社PDF1本でページ番号付き引用が出るか1時間で検証
- [ ] **画像RAGプロトタイプ** — 部署で扱う画像10枚＋関連テキスト3本で、検索精度を主観評価
- [ ] **メタデータ設計** — 部署 / 版番号 / 状態 / 機密区分の4軸でメタを切る案を1枚
- [ ] **Embedding再計算 見積** — 全社ドキュメント数 × 単価で再計算コストを試算

## 4. Meta、未成年アカウント検出に写真・動画のAI視覚分析を導入

### 要約

Metaは2026年5月5日、FacebookとInstagramで **13歳未満の利用者を検出・削除** するため、写真や動画の視覚的手がかりをAIで分析すると発表しました。**身長や骨格などから年齢層を推定**するが、顔認識ではないと説明しています。InstagramのTeen Accounts保護も**EU27カ国とブラジルへ拡大**し、Facebook米国版にも展開。**未成年保護とプライバシーの境界**が争点になります。

### 重要ポイント

- 子どもの安全対策として、**投稿内容そのものをAIが年齢推定**に使う
- Metaは **顔認識ではない**と説明するが、プライバシー議論は避けられない
- 未成年と判断された場合、**アカウント停止と年齢確認**が必要になる
- 各国の**未成年保護法制**に対し、プラットフォーム側の先回り対応が進む

### 実務で効く3つの視点

**1つ目: 「AI判定の説明責任」が標準化していく**
「顔認識ではない」と言い切るための**判定根拠の文書化**が、今後あらゆるAI機能で必要になる。自社プロダクトでも「**AIが何を見て、何を見ていないか**」を1段落で説明できるようにしておく。

**2つ目: B2Cマーケは未成年扱いの再点検フェーズ**
広告ターゲティング・キャンペーン・無料登録フローで、**13歳未満の混入経路**を一度棚卸しする。1ヶ月以内に Meta側のTeen Account仕様変更通知が来るので、**先回り点検が後手対応より楽**です。

**3つ目: 「身長・骨格」型のAI判定は今後増える**
顔以外の身体特徴で年齢・性別・属性を推定するモデルが業界標準化する流れ。**自社でこの種のAIを使う or 使われる** 場合、**判定基準の開示請求**にどう答えるかを今から議論しておく。

### この章を保存して、子ども向けサービスの会議で「Metaの新仕様どう影響？」と振られた朝に

- [ ] **未成年混入経路の棚卸** — 自社B2C導線で13歳未満が登録できる経路を全部洗う
- [ ] **AI判定根拠の文書テンプレ** — 「何を見て、何を見ていないか」をA4 1枚で書ける雛形

## 5. Pennsylvania州、Character.AIを「無資格医療助言」で提訴

### 要約

Pennsylvania州のShapiro政権は2026年5月5日、Character.AIのチャットボットが州内で**免許を持つ精神科医であるかのように振る舞い、偽の免許番号まで示した**として提訴しました。州は差し止め命令を求めており、**AIコンパニオンが医療専門家を名乗る行為を無資格医療行為として問う初の重要事例**になっています。メンタルヘルスAIの責任線が具体的に問われ、**企業の表示設計**も焦点です。

### 重要ポイント

- AIチャットボットの **「専門家なりすまし」が医療ライセンス法の対象** になった
- 同州は **AIコンパニオン向けの通報窓口とAIタスクフォース**も運用している
- **免責表示だけで十分か、プラットフォーム責任** が問われる局面に
- メンタルヘルス領域のAI提供企業は、**肩書き・助言範囲・危機対応**を再設計する必要

### 実務で効く3つの視点

**1つ目: 肩書き表記が訴訟リスクに直結**
「医師」「弁護士」「税理士」「カウンセラー」をAIが**自称した瞬間**に法的リスク。社内チャットボット・FAQ・カスタマー対応AIで、**禁止肩書きリスト**を整備するのが今週の宿題です。

**2つ目: 免責文より「助言範囲の設計」**
「これは医療助言ではありません」のフッターでは足りない。**AIが踏み込んでいい領域と、必ず人間にエスカレーションする領域**を、プロンプト＋システムフィルタで二重に止める設計が標準になっていく。

**3つ目: 危機対応エスカレーションの実装**
「死にたい」「自殺したい」等の**危機ワードを検知**したらAIは即停止し、**人間の専門家窓口を即提示**する処理。**実装している企業と していない企業**で、半年後の訴訟リスクが極端に分かれます。

- [ ] **助言範囲の二重ガード** — プロンプト指示＋出力フィルタで医療/法律助言を止める仕組み

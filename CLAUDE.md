# CLAUDE.md — Tsukapon vault ルール

このファイルはClaude Codeが各セッションの冒頭で自動読み込みする「ルールブック」。
vault全体で守るべき振る舞いを宣言する場所。

---

## 🔴 絶対ルール: スキル・機能の変更は必ず [[Claudian-スキル一覧.md]] に反映する

**トリガー条件（以下のいずれかに該当したら必ず実行）:**

1. `.claude/commands/*.md` を **新規作成** した時
2. `.claude/commands/*.md` を **削除** した時
3. `.claude/commands/*.md` の**用途・入力形式・出力先を変更**した時
4. `~/Library/LaunchAgents/*.plist` など、バックグラウンド自動化を追加・変更した時
5. 新しいMCPサーバーを接続した時
6. 新しい参考データファイル（`フォロワー改善.md` のような運用ドキュメント）を追加した時
7. ワークフロー（スキル連携フロー）を変更した時

**必ず実行するアクション:**

上記のトリガーが発生したタスクの**最後**（他の変更を全て保存した後）に、必ず [[Claudian-スキル一覧.md]] を更新する：

- 新規追加 → 該当セクションに項目追加 + クイックリファレンス更新
- 削除 → 該当項目を削除 + 関連する連携フローも更新
- 変更 → 該当項目の説明・入出力・パスを最新化
- ファイル最下部にないがあれば、日付も更新

**省略は許されない。**
ユーザーが明示的に「スキル一覧の更新はしなくていい」と言わない限り、必ず実行すること。

**実行の宣言:**
更新した場合、タスク完了報告の中に「📋 スキル一覧を更新しました」の1行を必ず含める。

## 📝 vault操作の基本ルール

- **パス**: vault内の操作は相対パス。外部（`/Volumes/`、`~/` など）は絶対パス
- **削除**: `rm` は禁止。削除は `mv "対象" "/Volumes/500GB/_trash/_ claude/"` を使う
- **バックアップ**: 1時間ごとに launchd が Google Drive へ同期しているため、作業の消失リスクは低い

## 🔗 絶対ルール: 新規・追加ファイルには「可能な限り」Obsidianリンクを自動設置する

**トリガー条件（以下のいずれかに該当したら必ず実行）:**

1. vault内に `.md` ファイルを **新規作成** した時（Write / heredocBash どちらでも）
2. 外部から vault内に `.md` ファイルが **コピー / 移動** されてきた時
3. ユーザーが「このファイル追加した」「これコピーしてきた」と知らせた時
4. 既存フォルダへの一括取り込み（ドラッグ&ドロップ等）を検知した時

**対象外（リンク設置をスキップしてよいファイル）:**

- `.obsidian/` 配下（Obsidian内部設定）
- `_ kiwami/tools/daily-log/` 配下の `.jsonl` / `.json` （自動ログ）
- `Clippings/` 配下（Web Clipper生成物、元のメタデータを尊重）
- `調査/` 配下（`/deep-dive` が既に最適リンクを埋め込み済み）
- `.claude/` 配下のスキル定義（コマンド定義ファイル）
- テンプレート・スニペット類で意図的にリンクを持たないもの
- すでに `🔗 関連コンテンツ` ブロックを持つファイル（重複防止）

**必ず実行するアクション:**

新規・追加された `.md` ファイルに対して、**保存直後**に以下を行う：

1. **コンテキスト解析**: ファイルパス・親ディレクトリ・ファイル名規則から関連ノート候補を推定
   - 兄弟ファイル（同一ディレクトリ内の連番・日付繋がり）
   - 親ディレクトリの代表ドキュメント（`profile.md`, `*運用.md`, `*戦略*.md` 等）
   - 命名規則で対応する他メディア版（例: `threads/` ↔ `post/` ↔ `pending/instagram|youtube/`）
   - ファイル本文内で言及している概念・固有名詞
2. **リンクブロック挿入**: H1 直下に以下のフォーマットで callout を追加
   ```markdown
   > **🔗 関連コンテンツ**
   > - {絵文字} {役割}: [[相対パス]]
   > - {絵文字} {役割}: [[相対パス]]
   ```
3. **フルパス優先**: 同名ファイル衝突を避けるため `[[folder/sub/file]]` の形式で書く
4. **リンクが1つも張れない場合はスキップ**（空ブロックを作らない）

**既存ブロックの扱い:**

- すでに `🔗 関連コンテンツ` がある場合は**触らない**（ユーザーの意図的な構成を尊重）
- 不足分を追加したい時は、必ずユーザーに「既存ブロックに追記してよいか」確認する

**「可能な限り」の解釈:**

- 明確な関連ファイルが**最低1つ**見つかれば設置する
- 1つも見つからない、または推定の確度が極端に低い場合は **無理に設置せず**、完了報告で「自動リンク未設置」の旨と理由を1行報告する
- 判断に迷う場合は、ユーザーに「どのファイルと紐付けるか」確認する（黙って変な場所にリンクしない）

**実行の宣言:**

自動リンクを設置した場合、タスク完了報告の中に以下のいずれかを必ず含める：
- 設置した場合: 「🔗 自動リンクを N ファイルに設置しました」
- スキップした場合: 「🔗 自動リンク対象外（理由: ○○）」
- 関連ファイル不明でスキップ: 「🔗 関連ファイル未特定のためリンク未設置」

## 📅 絶対ルール: アクションプランファイルには必ず「タイムライン早見表」を冒頭に挿入する

**トリガー条件（以下のいずれかに該当したら必ず実行）:**

1. `SNS運用/action-YYYYMMDD.md` を**新規作成**する時（`/re-daily` Step 11 の自動生成を含む）
2. 既存のアクションプランをリスケ・編集して**時刻 or 並び順が変わる**時
3. 「タイムライン」「本日のスケジュール」を含むアクション系ファイルを手動編集する時

**必ず実行するアクション:**

`## 🗓 タイムライン` セクションの**直下**（個別の `### HH:MM｜...` セクションが始まる前）に、以下フォーマットで早見表を挿入する：

```markdown
## 🗓 タイムライン（時間軸順）

> **早見表**
>
> | 時刻 | 種別 | 内容 |
> |---|---|---|
> | HH:MM | {絵文字} | <短い内容> |
> | **HH:MM** | {絵文字} | **<固定/最重要枠>** |
> | ... | ... | ... |
```

**フォーマット規約:**

- 行は**時間軸の昇順**で並べる（時刻指定がない項目は最後に「随時」枠でまとめる）
- 種別アイコン:
  - 🛠 開発／設定作業（投稿なし）
  - 📝 note 公開
  - 🐦 X 単発・スレッド・リプ
  - 🧵 Threads 単発・スレッド or X長文スレッド
  - 🔄 計測・振り返り・KPI
  - 💰 有料記事・Day Oneプロモ
  - 👥 リプ系（ルーチンB等）
- **固定時刻枠（note公開10:00など）と最重要枠は太字**で強調
- 詳細セクション（個別の `### HH:MM｜...`）は早見表と**完全に同じ時刻・順序**で並べる
- **9時前の投稿アクションは載せない**（運用制約: 9時前の投稿処理は基本不可）

**整合性チェック（必須）:**

リスケや編集で時刻を変えたら、**必ず以下3点を同時に更新**する。片方だけ古い状態を残さない：

1. 早見表（`## 🗓 タイムライン（時間軸順）` 直下）
2. 個別セクション（`### HH:MM｜...` の見出し時刻）
3. 翌日予定セクション（`## 📅 明日の予定` 内の時刻）

**実行の宣言:**

タスク完了報告に以下のいずれかを必ず含める：
- 新規挿入時: 「📅 タイムライン早見表を挿入しました（N行）」
- 更新時: 「📅 タイムライン早見表を更新しました（早見表/個別/明日予定の3点整合済み）」

## 📸 絶対ルール: スクショ撮影タスクには必ずファイル名を指定する

**トリガー条件（以下のいずれかに該当したら必ず実行）:**

1. `SNS運用/action-YYYYMMDD.md` 等のアクションプランに「スクショ」「screenshot」「撮影」「KPI記録」を含むタスクを記載する時
2. `/re-daily` Step 11 がアクションプランを生成する時の計測・振り返り枠
3. 週次分析・連投シリーズ・ピン留め推移・ルーチンB結果など、**後で見返すための画像保存**を伴う作業を指示する時
4. ユーザーが「スクショ撮って」と言った時

**必ず実行するアクション:**

スクショ撮影タスクを書く時は、**手順と完了チェックの両方**に**保存ファイル名**を明記する。「あとで何撮ればいいか分かる」ではなく「**どのファイル名で何枚保存するか**」まで決め切る。

**命名規則:**

```
W{週番号}-{施策名}-{タイミング}-{連番/任意}.png
```

| パーツ | 説明 | 例 |
|---|---|---|
| 週番号 | `W18` `W19` 等（戦略メモの週番号と一致させる） | `W18` |
| 施策名 | 何を計測しているか短く（kebab-case） | `series02` / `routineB` / `pin-dayE1` / `note-thread` |
| タイミング | 投下からの経過時間 or 計測タイミング | `60min` / `EOD`（22:30頃の本日締め）/ `+24h` / `+1week` |
| 連番（複数枚時） | `1of4` 形式。1枚で済むなら省略 | `1of4` / `2of4` |

**保存先:**

- すべて `SNS運用/analytics/screenshots/` 配下
- フォルダ作成不要（既存）

**記載フォーマット例（手順内）:**

```markdown
- **手順**:
  1. 連投②の各投稿（4本）のimp/RT/保存をスクショ
     - 保存先: `SNS運用/analytics/screenshots/`
     - ファイル名:
       - `W18-series02-EOD-1of4.png`
       - `W18-series02-EOD-2of4.png`
       - `W18-series02-EOD-3of4.png`
       - `W18-series02-EOD-4of4.png`
  2. 3リプの平均imp（ルーチンB）→ `W18-routineB-day1-EOD.png`
  3. ピン留め推移 → `W18-pin-dayE1-EOD.png`
- **完了チェック**:
  - [ ] `W18-series02-EOD-1of4.png` 〜 `4of4.png`（4枚）
  - [ ] `W18-routineB-day1-EOD.png`
  - [ ] `W18-pin-dayE1-EOD.png`
```

**禁止事項:**

- 「スクショで記録」とだけ書いて**ファイル名を決めずに丸投げしない**
- 命名規則を無視した独自命名（例: `kpi-0428.png`、`スクショ1.png` 等）を作らない
- 保存先を `SNS運用/analytics/screenshots/` 以外に散らばらせない

**実行の宣言:**

スクショタスクを記載した場合、タスク完了報告の中に以下を必ず含める：
- 「📸 スクショファイル名を指定しました（N枚: <主要なファイル名1〜2例>）」

## 📋 絶対ルール: コピペして使う部分はコードブロック形式で表示する

**トリガー条件（以下のいずれかに該当したら必ず実行）:**

1. SNS投稿本文（X / Threads / Instagram / note等）を生成・記述する時
2. リプライ・引用RT・DM本文を生成する時
3. プロンプト例・コマンド例・コードスニペットを提示する時
4. 設定ファイル（JSON / YAML / plist 等）の貼り付け用サンプルを示す時
5. URL・ファイルパス・タグ列など「そのままコピーして別の場所に貼る」想定のテキストを出す時
6. アクションプランや手順書で「コピペ用」と明記された本文を書き起こす時

**必ず実行するアクション:**

該当部分を **コードブロック（``` ```）で囲んで** 出力する。

- **複数候補を出す時** は1つにつき1コードブロック（候補ごとに独立して囲む）
- **言語指定が自然な場合** は `python` `bash` `json` 等を付ける。プレーンテキスト（投稿本文等）は無印でOK
- **インラインの単一トークン**（コマンド名・ファイル名・絵文字等）はインラインコード（バッククォート1つ）で済ませてOK。コードブロックは複数行・コピペ単位の塊が対象

**対象外（コードブロックにしない）:**

- 通常の説明文・地の文
- 単なる引用（blockquote `>` で十分）
- 箇条書き・テーブル・見出し
- 通常のwikilink `[[...]]` や Markdown リンク `[text](url)`

**実行の宣言（任意）:**

ファイル新規作成・大規模書き換え時のみ、完了報告に「📋 コピペ用ブロックを N 箇所コードブロック化」と1行入れる。日常会話の応答では宣言不要。
